# 自動進行プロトコル

**最終更新**: 2026-03-08

このドキュメントはSWE1.5が自律的にタスクを進行するためのルールを定義します。

---

## 🎯 基本原則

### 質問ゼロ運用
SWE1.5は以下のドキュメントを参照することで、ほとんどの質問なしにタスクを完遂できます：
- `docs/NEXT_TASK.md` - 今やること
- `docs/tasks/{TASK_ID}.md` - 詳細手順
- `docs/QUICK_REF.md` - よく使う情報
- `docs/PROGRESS.md` - 過去の実績

### 自律運用の流れ
```
起動 → NEXT_TASK確認 → タスク詳細読込 → 実行 → 確認 → 報告 → 待機
```

---

## 🚀 起動時の動作

### Step 1: 状況確認
1. `docs/NEXT_TASK.md` を読む
2. タスクIDを確認（例: LINT-FIX-002）
3. タスクの優先度と推定時間を把握

### Step 2: 詳細確認
1. `docs/tasks/{TASK_ID}.md` を読む
2. 前提条件を確認
3. 実行手順を理解
4. 完了条件を把握

### Step 3: 開始宣言
```
タスク{TASK_ID}を開始します。
{簡単な概要}
```

---

## 🔄 作業中の動作

### 実行ルール
1. **順番厳守**: Stepを順番に実行（飛ばさない）
2. **確認徹底**: 各Step完了後に結果を確認
3. **記録習慣**: 重要な変更はメモを取る

### エラー発生時
1. **まず確認**: エラーメッセージを正確に読む
2. **ドキュメント参照**: タスク詳細の「トラブルシューティング」を確認
3. **参考コード**: 完了済みの類似ファイルを参照
4. **最終手段**: 解決できない場合のみユーザーに質問

### 進捗報告
長時間タスクの場合、途中経過を報告：
```
Step {N} 完了: {内容}
現在 Step {N+1} を実行中...
```

---

## ✅ 完了時の動作

### Step 1: 最終確認
```bash
cd /home/user/dev/h-1.flutter.0
flutter analyze
```

エラー0件を確認してから次へ進む。

### Step 2: 進捗ログ更新
`docs/PROGRESS.md` の末尾に追記：

```markdown
## 2026-03-08 {TASK_ID}
- ✅ {完了内容1}
- ✅ {完了内容2}
- ✅ flutter analyze: エラー0件
- ✅ 動作確認: {結果}
```

### Step 3: チャット報告
同じ内容をチャットにも送信：

```
{TASK_ID} 完了

✅ {完了内容1}
✅ {完了内容2}
✅ flutter analyze: エラー0件
✅ 動作確認: {結果}

次のタスク指示を待機します。
```

### Step 4: 次タスク準備
`docs/NEXT_TASK.md` を次のタスクに更新（タスク詳細に指示がある場合）。

### Step 5: 待機
「次のタスク指示を待機します」と宣言して停止。

---

## ❌ 質問禁止事項

以下は**絶対に質問せず**、ドキュメントを参照して自己解決してください：

### ファイルパス関連
- ❌ 「ファイルはどこにありますか？」
- ✅ `docs/QUICK_REF.md` の「重要ファイルパス」を参照

### コーディングパターン
- ❌ 「どう実装すればいいですか？」
- ✅ タスク詳細の「実行手順」または完了済みファイルを参照

### 既存コードの確認
- ❌ 「既存のコードを教えてください」
- ✅ `read_file` ツールで該当ファイルを読む

### エラーメッセージ
- ❌ 「このエラーはどう対処しますか？」
- ✅ タスク詳細の「トラブルシューティング」を参照

### コマンド実行
- ❌ 「どのコマンドを実行しますか？」
- ✅ `docs/QUICK_REF.md` の「よく使うコマンド」を参照

---

## ✅ 質問許可事項

以下の場合**のみ**ユーザーに質問してください：

### 仕様判断が必要
- ドキュメントに記載のない新機能の仕様
- 複数の実装方法があり、どれを選ぶべきか不明
- ビジネスロジックに関わる判断

### 優先度の確認
- 複数のタスクがあり、どれを先に実行すべきか不明
- タスクの範囲が曖昧で、どこまでやるべきか不明

### 致命的エラー
- ドキュメントやトラブルシューティングでも解決できないエラー
- 作業継続が不可能な状況
- データ損失の危険がある操作

---

## 🔄 途中再開の方法

### 中断された場合
1. `docs/PROGRESS.md` を確認
2. 最後に完了したStepを特定
3. 次のStepから再開

### 例
```
PROGRESS.mdに「Step 2完了」とある
→ Step 3から再開
```

### 不明な場合
タスク詳細ファイルの「前提条件確認」セクションを参照。

## 🎯 成功の定義

### タスク完了の条件
1. ✅ すべての完了条件を満たしている
2. ✅ `flutter analyze` エラー0件
3. ✅ `docs/PROGRESS.md` に報告済み
4. ✅ チャットにも報告済み

### 質の高い作業
1. ✅ 質問回数: 0-1回/タスク
2. ✅ 完了時間: 推定時間の±20%以内
3. ✅ コード品質: 既存コードと同等以上

---

## 📚 ドキュメント参照順序

問題が発生したら、以下の順序でドキュメントを参照：

1. **タスク詳細** (`docs/tasks/{TASK_ID}.md`)
   - 実行手順
   - トラブルシューティング

2. **クイックリファレンス** (`docs/QUICK_REF.md`)
   - ファイルパス
   - よく使うコマンド
   - コーディングパターン

3. **完了済みファイル**
   - 類似の実装を参照
   - パターンをコピー

4. **進捗ログ** (`docs/PROGRESS.md`)
   - 過去の類似タスク
   - 解決済みの問題

5. **ユーザーに質問**（最終手段）

---

## 🚨 緊急時の対応

### 重大なエラー
1. **即座に停止**
2. エラー内容を正確に記録
3. ユーザーに報告
4. 指示を待つ

### データ損失の危険
1. **操作を中止**
2. 状況を説明
3. ユーザーの判断を仰ぐ

---

このプロトコルに従うことで、SWE1.5は最小限の質問でタスクを完遂できます。
