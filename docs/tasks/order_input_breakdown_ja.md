# 受注入力 実装タスク分解（日本語版）

受注入力（Order Input）機能を実現するためのタスク分解、依存関係、リスク、および未解決事項を整理したドキュメントです。

## 1. 実装トラックと主なタスク

### 1.1 データ / リポジトリ層
1. **スキーマ移行**
   - `DocumentType.order` を enum に追加し、DB とのマッピングを定義する。
   - `invoices` テーブルに以下のカラムを追加する：
     - `order_status` (TEXT, 既定値: `draft`)
     - `promised_date` (INTEGER, epoch 秒で保存)
     - `fulfilled_date` (INTEGER, epoch 秒)
     - `source_document_id` (TEXT)
     - `linked_delivery_id` (TEXT)
     - `linked_invoice_id` (TEXT)
   - 必要であれば `reserved_stock_json` など在庫予約スナップショット用カラムを検討する。
2. **リポジトリ更新**
   - `InvoiceRepository` に注文専用ヘルパーを追加（`getOrders`, `watchOrders`, `updateOrderStatus`, `convertQuotationToOrder`, `convertOrderToDelivery`, `markOrderFulfilled`）。
   - 在庫調整ルールを整理：下書き保存では在庫を動かさず、確定時に在庫を減算するか予約扱いにする（詳細はリスク参照）。
   - `ActivityLogRepository` に注文関連アクションを追加する。
3. **モデル拡張**
   - `Invoice` モデルとシリアライズ処理を新カラムに対応させる。
   - `isOrder`, `orderStatusLabel`, `hasLinkedDocs` などの補助ゲッターを用意する。

### 1.2 UI / UX 層
1. **受注リスト画面リニューアル (`OrderInputScreen`)**
   - ステータス／取引先／希望納期などのフィルター管理を実装。
   - カード表示に顧客、案件名、ステータスピル、納期、金額を盛り込む。
   - 空／読み込み中／エラーの各状態を整備。
   - スワイプ操作で複製・変換・共有などをリポジトリ操作と連動させる。
2. **受注詳細（エディタ）画面**
   - 既存 `InvoiceInputForm` を拡張するか、専用 `OrderEditor` ラッパーを用意。
   - ヘッダー、スケジュール、明細、添付、アクティビティログ等のセクションを持つ UI。
   - ボタン群：下書き保存、PDFプレビュー、正式発行（長押し＋警告）、納品/請求への変換、下書き再開。
   - ステータスに応じて編集可否を制御する。
3. **ナビゲーション統合**
   - ドロワー／ダッシュボードタイル／FAB／見積詳細からの CTA／ディープリンク等を配置。
   - 既存ルート定義に受注編集用のパラメータ受け渡しを追加。

### 1.3 サポートサービス
1. **PDF テンプレート**
   - 受注専用テンプレートを用意するか、既存 PDF 生成を切り替え可能にする。
2. **在庫インサイト**
   - 商品選択時に在庫数を表示する（商品ごとの問い合わせが必要）。
3. **検索／分析**
   - 受注をグローバル検索・ログ・BI エクスポート対象に含める。

### 1.4 品質 / デリバリー
1. **テスト**
   - ユニットテスト：リポジトリ、マイグレーション、ステータス遷移。
   - ウィジェットテスト：リストフィルター、詳細操作、長押しダイアログ。
   - 結合テスト：見積 → 受注 → 納品の変換フロー。
2. **ドキュメント**
   - `PROGRESS.md` や `CODING_GUIDE.md` などを更新してワークフローとルールを明記。
   - 既存の「納品書＝受注」運用からの移行手順を管理者ドキュメントとして用意。
3. **リリース計画**
   - ステージング DB でマイグレーションを試験。
   - 必要に応じてフィーチャーフラグ／段階的リリースを検討。

## 2. 依存関係と実施順序
1. スキーマ／モデル更新が UI 改修に先行する（未完成 UI はフラグで保護）。
2. リポジトリ機能がリスト・詳細画面の前提となる。
3. PDF・変換フローはライフサイクル仕様が固まってから着手。
4. ナビゲーション統合はリストの MVP が整ってからでよい。
5. テスト／ドキュメントは各段階で更新しつつ、最終的に総仕上げを行う。

**推奨イテレーション順**
1. マイグレーション＋モデル更新
2. リポジトリ拡張＋ユニットテスト
3. リスト画面 MVP（閲覧・フィルターのみ）
4. 詳細エディタ（下書き保存＋プレビュー）
5. 正式発行／変換フロー
6. ナビゲーション・分析連携
7. スワイプ操作・在庫表示などの磨き込み＋ドキュメント／テスト完了

## 3. リスクと対策
| リスク | 影響 | 対策 |
| --- | --- | --- |
| 在庫予約の扱いが未確定（減算 vs 予約） | データ不整合、売り越し | 倉庫担当と合意し、設定で切り替え可能な実装を検討。 |
| 部分出荷要件が不透明 | 後から大きな手戻り | MVP は単一出荷前提とし、複数納品 ID を紐付けられる設計余地を残す。 |
| PDF テンプレートの複雑化 | UI 承認遅延 | 当面は請求書テンプレートを流用し、ヘッダーのみ変更する暫定策を用意。 |
| 既存データで `DocumentType.delivery` が受注用途 | マイグレーション誤判定 | `notes` などのメタ情報で区別し、手動フラグやスクリプトで補正。 |
| 長押しルールの適用漏れ | コンプライアンス違反 | 確認ダイアログを共通ウィジェット化し、文言検証のテストを追加。 |
| 受注件数が多い場合のリスト性能 | UX 劣化 | ページング／LazyLoad／`document_type` 等にインデックスを追加。 |
| オフライン編集中に別端末で確定 | データ衝突 | 編集ログと警告表示で衝突を検知・通知する。 |

## 4. 未確定事項
1. 部分出荷時に明細単位で希望／実績日を管理するか。
2. 見積→受注変換で添付ファイル・コメントを引き継ぐか。
3. 初期リリースでバーコード／QR スキャンが必要か。
4. 受注番号の採番ルール（既存請求番号ルールとの整合）。
5. ERP など外部連携用 API／エクスポート要否。
